# 経営所得安定対策等 申請入力システム 要件定義書

作成日: 2026-06-05

## 1. 目的

水田政策見直しと令和9年度以降の電子申請試行に向け、農業者がWeb上で交付申請書・営農計画書等を作成し、地域農業再生協議会等が確認・補正・CSV出力できる入力システムを構築する。

紙申請が残る農業者については、手書きPDFまたはスキャンPDFをAIで読み取り、申請フォームの下書きとして保存できるようにする。最終提出・外部連携は、人による確認とシステム検証を通したデータのみを対象にする。

## 3. 対象ユーザー

| ユーザー | 主な操作 |
| --- | --- |
| 農業者・法人担当者 | 申請作成、途中保存、PDFアップロード、AI読取結果の確認、提出 |
| 代理申請者 | 委任された農業者の申請作成・補正・提出 |
| 地域農業再生協議会・JA等 | 受付、内容確認、差戻し、代理入力、CSV出力、紙申請のPDF取込 |
| 行政・事務局 | 利用状況確認、項目定義管理、モデル設定、監査ログ確認 |
| システム管理者 | テナント、権限、外部連携、APIキー、MCP/CSV出力設定 |

## 4. 全体方針

1. フロントエンドはレスポンシブWebアプリとし、スマートフォンでも入力可能にする。
2. バックエンドはConvexを前提とし、DB、サーバー関数、ファイルストレージ、外部AI呼び出しを統合する。
3. 入力項目は100から200項目規模になるため、巨大な1画面ではなく、申請ステップ・条件分岐・繰り返し明細で分割する。
4. すべての入力は下書きとして自動保存し、明示的な「下書き保存」も用意する。
5. AI-OCRは入力補助であり、AI結果だけで提出済みにしない。必ず人の確認・確定操作を通す。
6. 外部連携は当面CSV出力を主経路とし、将来のBAMS API公開またはMCP連携に備える。

## 5. 対象業務範囲

### 5.1 初期リリース対象

- 農業者による交付申請書A/B相当の作成
- 営農計画書・農地利用計画の作成
- 申請途中保存、再開、バージョン管理
- PDFアップロードからAI読取、下書き反映
- 地域協議会担当者による確認・差戻し・代理修正
- IF仕様に沿ったCSV出力
- エクスポート履歴・監査ログ

### 5.2 初期リリース対象外だが設計上考慮

- BAMSへの直接API送信
- 協議会システム各社との個別API連携
- 電子署名・マイナンバー連携
- 交付金額の最終算定・支払管理
- 全自治体向けのマスタ配信基盤

## 6. 申請ライフサイクル

| 状態 | 説明 | 操作可能者 |
| --- | --- | --- |
| `draft` | 農業者または代理者が編集中 | 農業者、代理申請者、協議会 |
| `ocr_draft` | PDF読取結果を下書き化した状態 | 協議会、農業者、代理申請者 |
| `needs_review` | 入力完了後、確認待ち | 農業者、協議会 |
| `returned` | 協議会から差戻し | 農業者、代理申請者 |
| `submitted` | 農業者側の提出完了 | 農業者、協議会 |
| `accepted` | 協議会側の受付完了 | 協議会 |
| `export_ready` | CSV出力可能 | 協議会、管理者 |
| `exported` | CSVまたはMCPで外部連携済み | 協議会、管理者 |
| `withdrawn` | 取下げ | 農業者、協議会 |

各状態遷移では、操作者、日時、変更差分、検証結果を監査ログに記録する。

## 7. 画面要件

### 7.1 農業者向け申請フォーム

入力画面は次のステップに分割する。

| ステップ | 主な内容 | 備考 |
| --- | --- | --- |
| 1. 申請者情報 | 氏名/法人名、住所、連絡先、経営形態、認定状況 | 別紙01、様式第1号A/B |
| 2. 申請する制度 | 水田活用、コメ新市場、畑作物、畑地化、ゲタ、ナラシ等 | 選択内容に応じて後続項目を表示 |
| 3. 口座・代理申請 | 口座届出、金融機関、ゆうちょ、代理申請区分 | 口座とゆうちょは同時入力不可 |
| 4. ゲタ・ナラシ確認 | 収入保険、税務申告、営農開始期間等 | ゲタ/ナラシ申請時のみ重点表示 |
| 5. 営農計画 | 共済加入、水稲用途別作付面積・数量 | 別紙12 |
| 6. 農地利用計画 | 耕地番号、分筆番号、作期、作物、面積、交付対象判断 | 別紙13。明細テーブル入力 |
| 7. ナラシ情報 | 出荷者コード、生産予定面積、対象農産物等 | 別紙07/08。ナラシ申請時のみ |
| 8. 添付・確認 | PDF、添付資料、個人情報同意、誓約事項 | 提出前チェック |
| 9. 確認・提出 | エラー/警告一覧、提出 | 提出後は協議会受付へ |

### 7.2 UX要件

- 画面上の入力項目は一度に出しすぎず、制度選択・作物選択に応じて必要な項目だけを表示する。
- 各ステップに「保存済み」「未入力」「要確認」「エラーあり」の状態を表示する。
- 手動の「下書き保存」ボタンを常時表示し、自動保存の時刻も表示する。
- スマートフォンでは筆情報や面積入力をカード型または行追加型にし、横スクロール前提にしない。
- 前年データが存在する場合は「前年からコピー」を提供する。
- 長い項目名には補足ヘルプを付けるが、画面の説明文を増やしすぎず、必要時だけ開けるようにする。
- 数値項目には単位を明示する。例: `㎡`, `kg`, `kg/10a`, `人`
- 必須エラーだけでなく、CSV出力時に問題になる形式エラーを入力時点で示す。

| データ群 | ファイル識別コード | 項目数 | 必須項目数 | 主な内容 |
| --- | ---: | ---: | ---: | --- |
| 共通 申請情報 | `1` | 71 | 43 | 申請者、住所、経営形態、口座、申請メニュー、作付有無 |
| ナラシ 交付申請者情報 | `10` | 8 | 5 | 都道府県、県拠点、申請者管理コード、出荷者情報 |
| ナラシ 生産予定面積情報 | `11` | 6 | 6 | 対象農産物、地域等コード、生産予定面積 |
| 営農計画書 水稲用途別作付面積等 | `103` | 78 | 9 | 共済加入、水稲用途別の数量・面積、畑地化支援 |
| 営農計画書 農地利用計画（筆情報） | `104` | 86 | 11 | 耕地番号、分筆番号、作期、作物、面積、交付対象判断 |

### 8.1 項目辞書

Convex上に項目辞書を保持し、画面表示・入力検証・CSV出力で同一定義を参照する。

項目辞書は少なくとも以下を持つ。

| 属性 | 内容 |
| --- | --- |
| `fieldKey` | システム内部キー。例: `applicant.nameKana` |
| `sourceSheet` | 参照元IF仕様。例: `別紙01` |
| `csvColumnNo` | CSV出力順 |
| `csvName` | IF仕様上の項目名 |
| `label` | 画面表示名 |
| `dataType` | 文字列、数値、日付、真偽値、コード |
| `maxLength` | 文字数または桁数 |
| `requiredRule` | 常時必須、条件付き必須、任意 |
| `codeSet` | コード値定義。例: `1=申請する,0=申請しない` |
| `validationRules` | 未入力、桁数、禁則文字、日付、相関チェック |
| `exportTransform` | 0埋め、全角/半角変換、日付形式、Shift-JIS変換 |
| `displayCondition` | 制度選択・作物選択に応じた表示条件 |
| `ocrHint` | PDF読取時の探索ラベル、帳票位置、候補表現 |

## 9. バリデーション要件

### 9.1 単項目チェック

- 必須入力
- 型チェック: 文字列、数値、日付
- 桁数チェック
- 半角数字、半角英大文字、半角カナ、全角文字数字の制限
- 日付の存在チェック
- コード値チェック
- 禁則文字チェック
- 単位付き数値の整数桁数チェック

### 9.2 相関チェック

最低限、次を実装する。

- 都道府県コード、地域協議会コード、申請年産がファイル内で一致していること
- 交付申請者管理コードの重複判定
- 申請年産が画面で選択した年度と一致すること
- 金融機関口座情報とゆうちょ口座情報が同時入力されていないこと
- ゲタ/ナラシを申請する場合、担い手経営安定法対象や認定状況との矛盾を検出すること
- ナラシ申請時に、ナラシ交付申請者情報と生産予定面積情報が揃っていること
- 農地利用計画のキーは、申請年産、交付申請者管理コード、耕地番号、分筆番号、作期の組み合わせで一意であること
- 作物名、作物名（選択）、作物種別のいずれかがIF仕様の関連チェックに反しないこと
- 水稲用途別の合計面積と明細上の面積に大きな不整合がないこと

### 9.3 エラー区分

| 区分 | 意味 | 提出可否 |
| --- | --- | --- |
| エラー | IF仕様・必須条件に違反 | 不可 |
| 警告 | 提出は可能だが確認が必要 | 可 |
| OCR要確認 | AI読取結果の確信度が低い | 確認済みにするまで不可 |
| システム注意 | CSV変換、文字コード、外部連携で注意 | 原則可 |

## 10. PDF取込・AI読取要件

### 10.1 入力

- 対応ファイル: PDF、JPEG、PNG
- 想定PDF: 手書きスキャン、スマホ撮影PDF、記入済みExcelをPDF化したもの
- 最大ページ数・ファイルサイズはプロバイダごとに設定可能にする
- アップロードファイルはConvex File Storageまたは外部ストレージに保存し、申請IDに紐づける

### 10.2 処理フロー

1. ファイルアップロード
2. ウイルススキャンまたはMIME検証
3. PDFをページ単位に分割
4. 傾き補正、回転補正、解像度確認
5. 帳票種別判定: 様式第1号A/B、様式第2号、別紙、その他
6. AIモデルに項目辞書と対象ページを渡して構造化抽出
7. 抽出結果をJSONとして保存
8. 既存下書きとの差分を作成
9. 人が確認し、採用・修正・却下
10. 採用済み項目だけ申請下書きへ反映

### 10.3 モデル切替

モデルは固定実装せず、プロバイダ抽象化を行う。

| プロバイダ | 用途 | 備考 |
| --- | --- | --- |
| OpenAI | 画像/PDFからの構造化抽出、JSON Schema出力 | Responses APIの画像入力・Structured Outputsを候補 |
| Anthropic | PDF全体理解、手書き・表の読み取り比較 | PDFをテキストと画像の両方で処理可能 |
| Google Gemini | PDFドキュメント理解、表・長文PDFの比較検証 | PDFから構造化出力する候補 |

設定項目:

- `provider`
- `modelId`
- `displayName`
- `enabled`
- `defaultForFormType`
- `supportsPdf`
- `supportsImage`
- `supportsJsonSchema`
- `maxPages`
- `maxFileSizeMb`
- `temperature`
- `costLimitPerJob`
- `timeoutSeconds`

### 10.4 抽出JSON形式

AIの出力は、直接フォーム値として扱わず、候補値として保存する。

```json
{
  "documentType": "form_1_ab",
  "model": {
    "provider": "openai",
    "modelId": "configured-model-id"
  },
  "fields": [
    {
      "fieldKey": "applicant.nameKanji",
      "label": "交付申請者名（漢字）",
      "value": "農林 太郎",
      "confidence": 0.92,
      "source": {
        "page": 1,
        "bbox": [100, 220, 460, 260],
        "rawText": "農林 太郎"
      },
      "status": "suggested"
    }
  ],
  "tables": [
    {
      "tableKey": "landParcels",
      "rows": []
    }
  ],
  "warnings": []
}
```

### 10.5 人手確認UI

- PDF画像と入力フォームを左右または上下に並べる
- AIが埋めた項目は色で区別する
- 信頼度が低い項目は「要確認」として一覧表示する
- 採用、修正して採用、却下、空欄のまま、を選べる
- 「下書き保存」ボタンでOCR結果を申請下書きに保存する
- 監査用に、AIの候補値、採用後の値、採用者、日時を保持する

## 12. CSV出力要件

### 12.1 出力単位

IF仕様に合わせ、次のCSVを出力する。

| CSV | ファイル識別コード | レコード単位 |
| --- | ---: | --- |
| 共通 申請情報 | `1` | 申請者1件につき1レコード |
| ナラシ 交付申請者情報 | `10` | ナラシ申請者1件につき1レコード |
| ナラシ 生産予定面積情報 | `11` | 対象農産物・地域等コード単位 |
| 営農計画書 水稲用途別作付面積等 | `103` | 申請者1件につき1レコード |
| 営農計画書 農地利用計画（筆情報） | `104` | 筆・作期単位 |

### 12.2 形式

- 文字コード: Shift-JIS
- 囲み文字: ダブルクォーテーション
- 改行コード: LFまたはCRLFを設定可能
- 項目順: IF仕様のNo.順
- 日付形式: `YYYY/MM/DD`
- コード値はIF仕様に合わせる
- 不足桁の0埋めはIF仕様の備考に従う
- 出力前に必須・形式・相関チェックを再実行する

### 12.3 出力操作

- 協議会単位、申請年産単位、状態単位で絞り込み出力できる
- 出力前に件数、エラー件数、警告件数を表示する
- 出力済み申請はロックまたは再出力扱いにする
- ZIPで複数CSVと出力レポートをまとめる
- 出力履歴には対象条件、件数、実行者、日時、ファイルハッシュを保存する

## 13. MCP/API連携要件

将来的なBAMS API公開または協議会システム連携に備え、MCPまたはHTTP APIで以下の操作を提供できる設計にする。

| ツール/API | 用途 |
| --- | --- |
| `list_applications` | 申請一覧取得 |
| `get_application` | 申請詳細取得 |
| `validate_application` | 検証実行 |
| `export_csv` | CSV生成 |
| `create_draft_from_pdf` | PDFから下書き作成 |
| `update_application_field` | 外部システムからの補正 |
| `get_export_status` | 出力ジョブ状態確認 |

MCP/APIには組織単位のスコープ、申請年産スコープ、読み取り/書き込み権限、レート制限、監査ログを必須とする。

## 14. セキュリティ・個人情報要件

- 個人情報、口座情報、農地情報、申請情報を高機密情報として扱う
- 組織・ロール単位のアクセス制御を実装する
- 農業者は自分または委任先の申請のみ閲覧可能
- 協議会担当者は所属組織の対象申請のみ閲覧可能
- AI APIに送信するデータは最小限とし、送信内容・モデル・日時を記録する
- APIキーはサーバー側のみで管理し、クライアントへ露出しない
- 添付PDF・OCR結果・CSV出力ファイルはアクセス権確認付きで配信する
- 監査ログは通常ユーザーが改ざんできない形で保持する
- データ保持期間、削除手順、AIプロバイダへの送信可否を運用規程として定義する

## 15. 非機能要件

| 項目 | 要件 |
| --- | --- |
| 可用性 | 申請期間中の平日日中は安定稼働。障害時は下書き喪失を防ぐ |
| 性能 | 通常入力の保存は1秒程度を目標。CSV出力・OCRは非同期ジョブ化 |
| 同時利用 | 初期は試行導入協議会単位の利用を想定し、段階的拡大可能にする |
| 互換性 | 最新のChrome、Edge、Safari。スマートフォンSafari/Chrome |
| アクセシビリティ | キーボード操作、ラベル、エラーメッセージ、色以外の状態表示 |
| 監視 | OCR失敗率、CSV出力失敗率、保存失敗、APIエラーを監視 |
| バックアップ | 申請データ、項目辞書、添付資料、出力履歴を復元可能にする |
| 保守性 | 年度・様式・IF仕様の差し替えをコード修正最小で行える |

## 16. MVP要件

### 16.1 MVPで必須

- Convex上で申請の作成、途中保存、再開ができる
- 別紙01、12、13の主要項目を画面入力できる
- 筆情報を複数行で登録できる
- PDFをアップロードし、モデルを選択してAI読取を実行できる
- AI読取結果を人が確認し、下書きへ反映できる
- 申請単位で検証エラーを一覧表示できる
- IF仕様に沿ったCSVを生成できる
- 協議会担当者が申請一覧、詳細、差戻し、CSV出力を行える

### 16.2 MVP後に拡張

- 全項目辞書の管理画面
- 前年データコピー
- 協議会システムごとのCSV変換プロファイル
- MCP連携の本番運用
- AIモデル比較評価画面
- 帳票PDF再生成
- 地図・筆ポリゴン連携

## 17. 受入条件

1. 申請者1件について、下書き保存、再開、提出、差戻し、再提出、受付、CSV出力まで通せる。
2. PDFから読み取った値は、AI候補として保存され、人が採用した項目だけフォーム値に反映される。
3. IF仕様の必須・型・桁数・コード値エラーが提出前に検出される。
4. CSVはファイル識別コード、項目順、文字コード、日付形式、0埋め規則に従って出力される。
5. すべての状態変更、OCR採用、CSV出力が監査ログに残る。
6. スマートフォン幅でも主要入力、下書き保存、エラー確認が破綻しない。
7. AIプロバイダを設定で切り替えられる。
8. 令和9年度仕様変更に対して、項目辞書とバリデーション定義を更新して追随できる。
